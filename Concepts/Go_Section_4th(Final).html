Topic 14

Mutexes & Synchronization

While channels are great for passing data between goroutines, sometimes you just need multiple goroutines to safely read and write to the same map or struct in memory. That's where sync.Mutex (Mutual Exclusion) comes in.

mutex.go
type AgentCache struct {
    mu     sync.Mutex
    status map[string]string
}

func (c *AgentCache) SetStatus(id, state string) {
    c.mu.Lock()
    defer c.mu.Unlock() // Always defer Unlock to ensure it runs even if we panic
    
    c.status[id] = state
}

func (c *AgentCache) GetStatus(id string) string {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    return c.status[id]
}

Go also provides sync.RWMutex. It allows multiple readers to hold the lock simultaneously, but only one writer. This is significantly faster if your data is read 99% of the time and rarely written to.

Interview Trap

You must never copy a Mutex (e.g., passing the struct by value). A Mutex is a stateful object. If you copy it, the copy gets its own internal lock state, entirely defeating the purpose of synchronization. Always pass structs containing Mutexes by pointer.

Interview Questions — Mutexes

QWhen should you use a Mutex vs. a Channel?⌄
Use a Mutex when you need to protect shared state or caches (e.g., updating a map). Use a Channel when you need to orchestrate execution, pass ownership of data, or create a pipeline between goroutines.
QWhat is the difference between sync.Mutex and sync.RWMutex?⌄
A standard Mutex locks completely—only one goroutine can read or write at a time. An RWMutex has RLock() and Lock(). Multiple goroutines can hold an RLock (read lock) simultaneously, but a Lock (write lock) requires exclusive access, blocking all new and existing readers.
QWhat happens if you call Unlock() on an already unlocked Mutex?⌄
It causes a fatal runtime panic. You cannot unlock a mutex that isn't locked. This is why defer mu.Unlock() immediately following mu.Lock() is the standard idiom—it perfectly pairs them.
QCan you lock a Mutex in one goroutine and unlock it in another?⌄
Technically yes, the Go runtime does not tie a Mutex to a specific goroutine ID (unlike thread-local locks in some languages). However, this is universally considered terrible practice and leads to unmaintainable code. If you need to lock in one place and unlock in another, you almost certainly need a Channel or a WaitGroup instead.
Topic 15

Defer, Panic, and Recover

Go doesn't have try/catch/finally. It handles expected failures via return values (error) and catastrophic, unrecoverable failures via panic. To clean up resources regardless of how a function exits, Go uses defer.

defer_eval.go
func AnalyzeSpatialData() {
    defer fmt.Println("1. This runs LAST (LIFO)")
    defer fmt.Println("2. This runs FIRST")
    
    fmt.Println("3. Doing spatial analysis...")
}
// Output:
// 3. Doing spatial analysis...
// 2. This runs FIRST
// 1. This runs LAST (LIFO)
Good to know

A deferred function's arguments are evaluated immediately when the defer statement is executed, not when the function actually runs at the end. For example, in defer fmt.Println(time.Now()), the time printed will be the time the defer was declared, not the time the function exited.

recover.go
func SafeWorker() {
    defer func() {
        // recover() stops the panic unwinding and returns the panic value
        if r := recover(); r != nil {
            fmt.Println("Recovered from panic:", r)
        }
    }()

    fmt.Println("Starting unsafe operation...")
    panic("catastrophic memory fault")
    fmt.Println("This line will never print")
}

Interview Questions — Defer & Panic

QIn what order do multiple defer statements execute?⌄
LIFO (Last In, First Out). The last defer statement pushed onto the stack is the first one pulled off when the function returns.
QCan a deferred function modify named return values?⌄
Yes! If a function has a named return value (e.g., func work() (err error)), an anonymous deferred closure can read and modify that err variable before the function actually hands it back to the caller. This is often used to wrap errors at the very end of a function.
QHow does recover() work, and where must it be called?⌄
recover() catches a panicking goroutine, stops the crashing sequence, and returns the value that was originally passed to panic(). It must be called directly inside a deferred function. Calling it anywhere else will just return nil and fail to stop the panic.
QDoes a panic in one goroutine affect others?⌄
Yes. An unrecovered panic in any goroutine will crash the entire Go program immediately. You cannot recover from a panic that occurred in a different goroutine.
Topic 16

Advanced Error Handling

Since Go 1.13, errors can "wrap" other errors to provide context (like a stack trace of strings) without losing the original error's type. This replaces the old habit of string-concatenating errors.

errors.go
var ErrTimeout = errors.New("provider timeout")

func DispatchToProvider(id string) error {
    err := pingProvider(id)
    if err != nil {
        // Use %w to WRAP the error, keeping ErrTimeout intact underneath
        return fmt.Errorf("gocatalyst dispatch failed: %w", err)
    }
    return nil
}

func main() {
    err := DispatchToProvider("groq-alpha")
    
    // errors.Is unwraps the chain and checks if ErrTimeout is anywhere inside it
    if errors.Is(err, ErrTimeout) {
        fmt.Println("Provider timed out, failing over...")
    }
}
Good to know

Use errors.Is(err, target) to check if an error matches a specific sentinel error value (like ErrTimeout). Use errors.As(err, &targetStruct) to check if an error is of a specific type (custom struct) and extract its fields.

Interview Questions — Errors

QWhat is the difference between %v and %w in fmt.Errorf?⌄
%v simply formats the error as a string, permanently losing the original error's type. %w wraps the error, creating an error chain. This allows the caller to unwrap it later using errors.Is or errors.As to inspect the root cause.
QWhy use errors.Is(err, ErrFoo) instead of err == ErrFoo?⌄
If the error was wrapped at any point in the call stack (e.g., fmt.Errorf("context: %w", ErrFoo)), the direct equality check err == ErrFoo will evaluate to false because the types no longer match exactly. errors.Is unwraps the error layer by layer, checking equality at every level.
QHow do you extract data from a custom error struct?⌄
Using errors.As. You pass the error and a pointer to a variable of your custom error type. If the error (or any wrapped error inside it) matches that type, errors.As populates your variable and returns true.
Topic 17

The Context Package

context.Context is the standard way to pass deadlines, cancellation signals, and request-scoped metadata (like trace IDs) across API boundaries and deep into goroutines.

context.go
// Context is always passed explicitly as the FIRST parameter
func FetchCableTelemetry(ctx context.Context, nodeID string) (Data, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET", "http://api/vigil/"+nodeID, nil)
    
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return nil, err // Will return context.Canceled if the timeout fired
    }
    defer resp.Body.Close()
    // ...
}

func main() {
    // Create a context with a strict 2-second timeout
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    
    // ALWAYS defer cancel to avoid background timer leaks
    defer cancel() 

    data, err := FetchCableTelemetry(ctx, "node-alpha")
}
Interview Trap

Never store a Context in a struct type (e.g., type Client struct { ctx context.Context }). Contexts should flow through the program as function arguments. Storing them in structs leads to confusing lifecycles where a struct instance becomes permanently cancelled.

Interview Questions — Context

QWhat are the golden rules of passing Context?⌄
1. Never store Context in a struct; pass it explicitly.
2. It should always be the first parameter of a function, traditionally named ctx.
3. Never pass a nil Context. If you aren't sure what to pass yet, use context.TODO().
QWhy must you always call the cancel() function returned by WithTimeout?⌄
When you create a context with a timeout, a background timer is allocated in memory. If the function finishes quickly but you don't call cancel(), that timer and its associated resources linger in memory until the timeout naturally fires, causing a memory leak. Deferring cancel() immediately frees those resources upon exit.
QShould you use context.WithValue to pass optional function parameters?⌄
Absolutely not. context.WithValue should strictly be used for request-scoped data that transits boundaries, like auth tokens, trace IDs, or request origin IPs. Using it as a "garbage bin" for optional configuration parameters bypasses compile-time type safety and makes the function's dependencies completely invisible.
QIf a parent context is cancelled, what happens to contexts derived from it?⌄
All children contexts (and their children) are immediately cancelled as well. Cancellation propagates down the tree, but never up.
Topic 18

Generics (Go 1.18+)

Before Go 1.18, writing a function that worked on any type required using interface{} and dangerous runtime type assertions. Generics introduce compile-time type parameters via square brackets [T].

generics.go
// K must be a comparable type (usable as a map key)
// V can be absolutely anything (any)
func GetMapKeys[K comparable, V any](m map[K]V) []K {
    keys := make([]K, 0, len(m))
    for k := range m {
        keys = append(keys, k)
    }
    return keys
}

func main() {
    scores := map[string]int{"Mujtaba": 95, "Usman": 88}
    
    // The compiler infers [string, int] automatically
    keys := GetMapKeys(scores) 
    fmt.Println(keys) // ["Mujtaba", "Usman"]
}

You can also create generic structs, which is perfect for building custom data structures or API response wrappers.

generic_struct.go
type Response[T any] struct {
    Data  T      `json:"data"`
    Error string `json:"error,omitempty"`
}

// Usage:
var r Response[int]
r.Data = 42

Interview Questions — Generics

QWhat is the comparable constraint?⌄
It is a built-in generic constraint that restricts a type parameter to only types that support the == and != operators (like strings, ints, and pointers). This is mandatory if you intend to use the generic type as a map key or compare instances directly inside the function.
QWhat is the difference between [T any] and using interface{} parameters?⌄
Using interface{} requires runtime type assertions to manipulate the underlying data, permanently losing compile-time safety. Generics ([T any]) are resolved at compile-time, ensuring full type safety and eliminating the need to cast the results back to their original type.
QWhen should you NOT use generics?⌄
If you just need to call a method on a value, use an Interface. Generics are for when the specific type of the data matters to the function's input/output mapping (e.g., returning a slice of the exact same type that was passed in), or for building agnostic data structures like Sets and Trees.
