Go concurrency, from first principles

Goroutines, channels, and the wait group that ties them together.

One reference for how Go's three core concurrency primitives actually behave underneath, how they compose into real patterns, and the questions interviewers reach for when they want to know if you've actually used them.

go func() make(chan T) sync.WaitGroup select { }
The short version

Three primitives, three jobs

Before the details, it helps to know what each piece is actually for. None of them overlap — each solves a different problem, and most real Go code uses all three together.

Goroutine — the worker

A function running independently, scheduled by the Go runtime instead of the OS. Cheap enough to start thousands of them without thinking twice.

Channel — the conveyor

A typed pipe that lets goroutines hand values to each other safely, without touching shared memory directly.

WaitGroup — the checkpoint

A counter that blocks one goroutine until a set of others have all finished. It carries no data — only "are we done yet?"

01 — Goroutines

A lightweight thread the runtime owns, not the OS

A goroutine is a function that runs concurrently with the rest of your program. You start one by putting go in front of a function call — the runtime takes it from there.

Under the hood, Go's scheduler multiplexes many goroutines onto a much smaller number of OS threads. This is usually called the GMP model: a Goroutine is a unit of work, an M is an OS thread ("machine") that actually executes code, and a P is a logical processor that holds a queue of runnable goroutines and the resources an M needs to run them. The scheduler hands goroutines to Ps, Ps feed Ms, and when an M blocks on a slow syscall, its P can be handed to a different M so other goroutines keep making progress. That's the mechanism that lets one process comfortably run hundreds of thousands of goroutines on just a handful of threads.

main.go
package main

import (
	"fmt"
	"time"
)

func sayHello() {
	fmt.Println("hello from a goroutine")
}

func main() {
	go sayHello()                     // scheduled — not guaranteed to have run yet
	fmt.Println("hello from main")
	time.Sleep(50 * time.Millisecond) // give the scheduler a chance to run it
}

Without that Sleep, main can return before sayHello ever gets scheduled — a goroutine is never automatically "joined" the way a thread might be. That's the entire reason WaitGroups and channels exist: something has to tell the main goroutine when the others are actually done.

Classic pitfall

Capturing a loop variable inside a goroutine closure is the single most common goroutine bug interviewers ask about. See the toggle below.

loopvar.go
var wg sync.WaitGroup
for i := 0; i < 3; i++ {
	wg.Add(1)
	go func() {
		defer wg.Done()
		fmt.Println(i) // on Go <1.22 this often prints 3,3,3
	}()
}
wg.Wait()

Before Go 1.22, every iteration of a for loop reused the same variable i. If the goroutines didn't get scheduled until the loop had already finished, they'd all read the final value. As of Go 1.22, each iteration gets its own fresh copy of i, so this specific version is actually safe on modern Go too — but you should still know the failure mode, because plenty of codebases run older Go and plenty of interviewers still ask about it.

loopvar_fixed.go
var wg sync.WaitGroup
for i := 0; i < 3; i++ {
	wg.Add(1)
	go func(i int) {       // i is now a fresh parameter, not shared state
		defer wg.Done()
		fmt.Println(i)
	}(i)
}
wg.Wait()

Passing i as a parameter (or writing i := i inside the loop body) makes the capture explicit and correct on every Go version, so most style guides still recommend it even though 1.22+ no longer strictly requires it.

02 — Channels

Typed pipes for goroutines to talk through

Go's guiding proverb is to not communicate by sharing memory, but instead share memory by communicating. A channel is that communication: a typed conduit you create with make, send into with <-, and receive from the same way.

Unbuffered

make(chan int) — a send blocks until another goroutine is ready to receive, and vice versa. The two sides rendezvous; there's no queue in between.

Buffered

make(chan int, 3) — a send only blocks once the buffer is full. It decouples sender and receiver timing, but it doesn't remove the need for coordination.

unbuffered.go
messages := make(chan string)

go func() {
	messages <- "ping" // blocks here until main receives
}()

msg := <-messages // blocks here until the goroutine sends
fmt.Println(msg)
buffered.go
queue := make(chan int, 3)
queue <- 1
queue <- 2
queue <- 3
fmt.Println(len(queue), cap(queue)) // 3 3 — len is items waiting, cap is buffer size
close(queue)

for v := range queue { // drains remaining buffered values, then exits
	fmt.Println(v)
}

Function signatures can restrict a channel to one direction, which documents intent and stops a consumer from accidentally sending:

direction.go
func producer(out chan<- int, n int) { // send-only from producer's point of view
	for i := 0; i < n; i++ {
		out <- i
	}
	close(out)
}

func consumer(in <-chan int) { // receive-only from consumer's point of view
	for v := range in {
		fmt.Println("got", v)
	}
}

Nil, closed, and open — the table interviewers actually want

This is the part of channels people get wrong under pressure. Memorize this table rather than the code:

OperationNil channelClosed channelOpen channel
Send ch<-vBlocks foreverPanics: send on closed channelBlocks until a receiver / buffer slot is ready
Receive <-chBlocks foreverReturns the zero value immediately, ok is falseBlocks until a value arrives
close(ch)Panics: close of nil channelPanics: close of closed channelCloses it — no error

A blocking nil channel isn't a bug — it's actually useful. Leaving a select case pointed at a nil channel permanently disables that case without an if branch, which is a clean way to turn off a timeout or a data source conditionally.

select_timeout.go
select {
case res := <-resultCh:
	fmt.Println("result:", res)
case <-time.After(2 * time.Second):
	fmt.Println("timed out waiting for a result")
}

select waits on several channel operations at once. If more than one is ready, it picks one at random — not in source order — and a default case makes the whole statement non-blocking by firing immediately when nothing else is ready. One subtlety with time.After: each call allocates a new timer that isn't garbage-collected until it fires, so calling it repeatedly inside a hot loop is a minor leak — context.WithTimeout is the safer choice in production code.

Rule of thumb

Only the sender should close a channel, and only one sender should ever do it. If several goroutines might write to the same channel, don't let any of them close it directly — have a separate coordinator close it once every writer has finished (the WaitGroup-then-close pattern in the Worker Pool section below is the standard way to do that safely).

03 — sync.WaitGroup

A counter that blocks until it hits zero

sync.WaitGroup doesn't carry any data — it only answers one question: has this specific set of goroutines finished yet? It's a counting semaphore with exactly three methods.

MethodWhat it does
Add(n)Increases the internal counter by n. Call it before starting the goroutines it accounts for.
Done()Decreases the counter by one. Equivalent to Add(-1) — almost always called via defer at the top of the goroutine.
Wait()Blocks the calling goroutine until the counter reaches zero.
waitgroup.go
func main() {
	var wg sync.WaitGroup
	urls := []string{"a.com", "b.com", "c.com"}

	for _, u := range urls {
		wg.Add(1)
		go func(u string) {
			defer wg.Done()
			fetch(u)
		}(u)
	}

	wg.Wait()
	fmt.Println("all fetches complete")
}
Two rules that prevent every common bug
  • Always pass a *sync.WaitGroup by pointer. Copying a WaitGroup copies its counter, so the copy and the original stop agreeing with each other — go vet will flag this for you.
  • Always call Add() before the go statement, never inside the goroutine. See the toggle below for why.
race.go
var wg sync.WaitGroup
for i := 0; i < 3; i++ {
	go func() {
		wg.Add(1)      // racing against Wait() below
		defer wg.Done()
		work()
	}()
}
wg.Wait() // may return immediately — counter could still be 0

There's no guarantee any goroutine has reached Add(1) before Wait() checks the counter. If the counter is still zero at that moment, Wait() returns immediately and the program moves on while goroutines are still running.

fixed.go
var wg sync.WaitGroup
for i := 0; i < 3; i++ {
	wg.Add(1)         // happens-before the goroutine even starts
	go func() {
		defer wg.Done()
		work()
	}()
}
wg.Wait() // guaranteed to see the full count

Calling Add(1) on the same goroutine that starts the worker, before the go statement, guarantees the counter is correct the instant Wait() is called.

One more edge case worth knowing: if a goroutine panics before its deferred Done() runs, the deferred call still executes during panic unwinding — but an unrecovered panic in any goroutine terminates the whole process regardless, so the WaitGroup's state stops mattering at that point.

04 — Putting it together

The worker pool: the pattern interviewers actually want to see

This is the canonical example that uses all three primitives at once, and it's the pattern most "design a concurrent X" interview questions are secretly asking for.

worker_pool.go
func main() {
	const numWorkers = 3
	jobs := make(chan int, 10)
	results := make(chan int, 10)
	var wg sync.WaitGroup

	for w := 1; w <= numWorkers; w++ {
		wg.Add(1)
		go worker(w, jobs, results, &wg)
	}

	for j := 1; j <= 9; j++ {
		jobs <- j
	}
	close(jobs) // no more work — each worker's range loop will end once drained

	go func() {
		wg.Wait()       // wait until every worker has exited its range loop
		close(results)  // only now is it safe to close results
	}()

	for r := range results {
		fmt.Println("result:", r)
	}
}

func worker(id int, jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {
	defer wg.Done()
	for j := range jobs {
		results <- j * j
	}
}

Notice the shape: jobs is closed by the single goroutine that produces work, results is closed by a dedicated goroutine that waits for every worker first — never by the workers themselves. That's the "one coordinator closes it" rule from the Channels section, applied.

Adding cancellation with context

A WaitGroup alone can't cut work short. When you need to stop early — a client disconnects, a timeout fires — pair channels with context.Context instead of, or alongside, a WaitGroup.

graceful_shutdown.go
func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
	defer cancel()

	results := make(chan int)
	go worker(ctx, results)

	select {
	case r := <-results:
		fmt.Println("got:", r)
	case <-ctx.Done():
		fmt.Println("cancelled:", ctx.Err())
	}
}

func worker(ctx context.Context, results chan<- int) {
	select {
	case <-time.After(5 * time.Second): // simulate slow work
		results <- 42
	case <-ctx.Done():
		return // stop early — no goroutine leak
	}
}
05 — Patterns & pitfalls

The shapes these primitives keep forming

Most concurrent Go code is a variation on a small handful of patterns. Recognizing them is most of what "systems design" questions in a Go interview are testing.

Fan-out, fan-in

Fan-out starts several goroutines to work on a shared input; fan-in merges several output channels into one. The merge step needs to know when every source is done before it can close the merged channel — which is exactly what a WaitGroup is for.

fanin.go
func fanIn(channels ...<-chan int) <-chan int {
	merged := make(chan int)
	var wg sync.WaitGroup
	wg.Add(len(channels))

	for _, c := range channels {
		go func(c <-chan int) {
			defer wg.Done()
			for v := range c {
				merged <- v
			}
		}(c)
	}

	go func() {
		wg.Wait()
		close(merged)
	}()

	return merged
}

Pipelines

Each stage is a function that takes an input channel and returns an output channel, run in its own goroutine. Stages compose by feeding one's output into the next's input, and closing propagates down the chain automatically once the source is exhausted and closed.

Goroutine leaks

A leaked goroutine is one that's still alive but will never finish — usually blocked forever on a channel nobody will write to or close. The fix is almost always the same: give every long-running goroutine a way out, typically a ctx.Done() case in its select.

Deadlocks

A single unbuffered send with no receiver anywhere deadlocks immediately: ch := make(chan int); ch <- 1. The Go runtime can detect and report this when every goroutine is asleep, but if even one other goroutine is still doing something, it stays silent and you're on your own with a stuck program.

Finding races instead of guessing at them

Unsynchronized access to shared state — like two goroutines incrementing the same plain int — is a data race, and it can silently produce wrong answers instead of crashing. Don't eyeball code for this; build and run with the race detector instead:

shell
go run -race main.go
go test -race ./...

The fix is either a sync.Mutex around the shared value, a sync/atomic operation for simple counters, or restructuring the code so only one goroutine ever touches that piece of state and everyone else talks to it over a channel.

Related primitives worth knowing the names of

sync.Mutex / sync.RWMutex guard shared state directly instead of routing access through a channel. sync.Once runs something exactly one time — the standard way to write thread-safe lazy initialization. sync/atomic gives lock-free operations for simple counters. And golang.org/x/sync/errgroup is a WaitGroup that also propagates the first error and can cancel a shared context — it's what most production code reaches for instead of raw sync.WaitGroup once error handling matters.

06 — Interview prep

Questions worth being able to answer cold

Grouped by topic, colour-coded to match the sections above. Click a question to expand it, or filter down to one topic at a time.

01What is a goroutine, and how does it differ from an OS thread?

A goroutine is a function that runs concurrently, scheduled by the Go runtime rather than the OS kernel. It starts with a tiny (~2 KB) stack that grows as needed, whereas an OS thread has a large fixed stack and is created and scheduled by the kernel. That difference is why a Go program can run comfortably with hundreds of thousands of goroutines but would struggle with anywhere near that many OS threads.

02How does Go's scheduler map goroutines onto OS threads?

Through what's usually called the GMP model: Goroutines are units of work, Ms are OS threads that execute code, and Ps are logical processors that each hold a run queue of goroutines and the resources an M needs. The scheduler multiplexes many Gs over few Ms via the Ps, and can move a P to a different M when the current one blocks on a syscall — that's the mechanism that keeps other goroutines running even while one is stuck waiting on I/O.

03What happens to goroutines still running when main() returns?

They're killed immediately — Go does not wait for other goroutines to finish when main exits. This is exactly why WaitGroups and channels exist: without one, there's no guarantee a goroutine you launched ever completes before the program ends.

04Does go f() run concurrently the instant it's called?

No — it schedules the function to run, but the scheduler decides when. The calling goroutine keeps executing immediately afterward, and there's no guarantee about ordering between the two unless you add explicit synchronization such as a channel or a WaitGroup.

05What's the difference between concurrency and parallelism in Go?

Concurrency is about structuring a program as independently executing pieces — dealing with many things at once. Parallelism is actually running computations simultaneously on multiple cores. Goroutines give you concurrency by default; whether that concurrency turns into real parallelism depends on GOMAXPROCS and how many CPU cores are actually available.

06Why is a goroutine's stack so much smaller than a thread's, and what happens as it grows?

Goroutine stacks start around 2 KB because the runtime manages them directly and can resize them on demand — when a goroutine needs more space, the runtime allocates a larger stack, copies the contents over, and fixes up pointers. OS threads can't do this cheaply, so they're given a large fixed-size stack up front, which is a big part of why they're far more expensive to create in bulk.

07What is a channel, and why does Go favor it over shared memory and locks?

A channel is a typed conduit, created with make(chan T), that goroutines use to send and receive values safely. Go's guiding idea is to communicate by sharing memory less and passing ownership of data through channels instead, which sidesteps a whole category of race conditions that come from multiple goroutines touching the same variable directly.

08Buffered vs unbuffered channels — how does the blocking behavior differ?

An unbuffered channel (make(chan int)) has no storage — a send blocks until a receiver is ready and vice versa, so the two goroutines rendezvous. A buffered channel (make(chan int, n)) can hold up to n values, so a send only blocks once the buffer is full and a receive only blocks once it's empty. Buffering decouples timing but doesn't remove the need to eventually drain or close the channel correctly.

09What happens sending on a closed channel? Receiving from one? Closing a nil or already-closed channel?

Sending on a closed channel panics with send on closed channel. Receiving from a closed channel never blocks — it returns the zero value immediately with ok == false. Closing an already-closed channel panics (close of closed channel), and closing a nil channel also panics (close of nil channel). See the full table in the Channels section — it's worth memorizing outright.

10What's the difference between len(ch) and cap(ch)?

cap(ch) is the fixed buffer size the channel was created with. len(ch) is how many values are currently sitting in that buffer waiting to be received. For an unbuffered channel, cap is always 0.

11What are directional channel types used for?

chan<- T restricts a channel parameter to sending only, and <-chan T restricts it to receiving only. Any bidirectional chan T can be passed where a directional type is expected, but not the reverse — this documents intent in a function signature and lets the compiler stop a consumer from accidentally writing back into a producer's channel.

12What does select do when multiple cases are ready, and what does default change?

If several cases are ready at the same time, select picks one at random — it makes no promise about source order. A default case fires immediately whenever no other case is ready yet, which turns the whole statement into a non-blocking check instead of one that waits.

13How do you put a timeout on a channel receive, and what's the catch?

Wrap it in a select with a time.After(d) case. The catch: every call to time.After creates a new timer that lingers in memory until it fires, so using it inside a loop that runs many times is a slow leak. context.WithTimeout, cancelled with defer cancel(), is the safer default in production code.

14What are the three sync.WaitGroup methods, and what does each one do?

Add(n) increases an internal counter by n; Done() decreases it by one (equivalent to Add(-1)); Wait() blocks the calling goroutine until the counter reaches zero. It's a pure counting semaphore — it carries no data about the work itself.

15Why must Add() be called before starting the goroutine, not inside it?

If Add(1) runs inside the goroutine, there's a race against Wait(): if Wait() checks the counter before that goroutine has actually reached Add(1), it can see zero and return immediately while goroutines are still running. Calling Add() on the launching goroutine, before go, guarantees the counter is correct the moment Wait() is called.

16Why should a WaitGroup always be passed by pointer?

A sync.WaitGroup holds its counter as a value. Copying it — say, by passing it by value into a function — copies the counter too, so the copy and the original immediately stop agreeing on how many goroutines are outstanding. go vet will warn about this if it can see the copy happening.

17What happens if Done() is called more times than Add()?

The counter goes negative, which panics immediately with sync: negative WaitGroup counter. This usually means Done() is being called more times than a goroutine actually represents — a common cause is calling it once per loop iteration inside a goroutine that only had Add(1) called for it once, total.

18What happens if a goroutine panics before its deferred Done() runs?

Deferred calls still run during panic unwinding, so Done() does execute. But that mostly doesn't matter: an unrecovered panic in any single goroutine crashes the entire process, not just that goroutine, so the WaitGroup's counter never gets the chance to be relevant.

19WaitGroup vs Mutex vs channel — when do you reach for each?

Use a WaitGroup when you need to know that a known number of goroutines have all finished, with no data to collect. Use a Mutex when multiple goroutines need to read and write the same in-memory value safely. Use a channel when goroutines need to actually hand data or results to each other, not just signal completion. Many real programs use all three at once, each for the job it's actually suited to.

20How do you build a worker pool, and why close results only after wg.Wait()?

Start N goroutines that all range over a shared jobs channel and write to a shared results channel, tracked by one WaitGroup entry each. See the full worked example above. Closing results too early — before every worker has finished writing to it — risks a "send on closed channel" panic from a worker that's still running; waiting on the WaitGroup first guarantees every writer is done before the channel closes.

21What's a goroutine leak, and how do you prevent one?

A goroutine leak is a goroutine that's still alive but will never finish — usually blocked forever waiting on a channel nobody will ever write to, close, or read from. The fix is almost always to give any long-running goroutine an escape hatch, typically a ctx.Done() case alongside its other work in a select.

22How do you actually detect a data race in Go?

With the built-in race detector: go run -race main.go or go test -race ./.... It instruments memory accesses at runtime and reports the exact two goroutines and stack traces involved in an unsynchronized read/write, rather than making you guess from a wrong answer alone.

23What does a Go deadlock look like, and why doesn't the runtime always catch it?

The simplest case is an unbuffered send with no receiver anywhere: ch := make(chan int); ch <- 1 blocks forever. The Go runtime detects and reports fatal error: all goroutines are asleep - deadlock! — but only when every single goroutine in the program is blocked. If even one other goroutine is still doing something, the program just hangs silently with no diagnostic at all.

24Explain the classic loop-variable capture bug — how did Go 1.22 change it?

Before Go 1.22, a for loop reused a single variable across every iteration, so a goroutine closure referencing it could see whatever value the loop had reached by the time the goroutine actually ran — often the final value for every goroutine. Go 1.22 changed the language so each iteration gets its own copy of the loop variable, which fixes this specific bug by default. It's still good style to pass the value in explicitly, both for clarity and for code that has to run on older Go.

25Why can concurrent fmt.Println calls produce garbled or interleaved output?

Each individual Println call is safe, but nothing coordinates the order in which different goroutines' calls happen, so their output can interleave line by line, or even mid-line under heavy load. It's a scheduling ordering issue rather than memory corruption — if you need clean sequential logs from concurrent code, route the log lines through a single channel and print them from one goroutine.

26How would you cancel a group of in-flight goroutines early?

The idiomatic way is context.Context: pass a ctx created with context.WithCancel or context.WithTimeout into every goroutine, and have each one select on ctx.Done() alongside its normal work so it can exit the moment cancellation fires. A hand-rolled "done channel" that's closed once achieves the same thing and predates context, but context also carries deadlines and request-scoped values, which is why it's the standard choice today.
